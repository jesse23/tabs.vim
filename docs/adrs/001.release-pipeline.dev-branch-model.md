# 001. Dev Branch Model

**SPEC:** release-pipeline  
**Status:** Accepted  
**Last Updated:** 2026-04-03

---

## Decision

Use `dev` as the integration branch for all development. Feature branches are cut from `dev` and merged back into `dev` via PR. CI publishes from `dev` to `main` automatically on every merge.

`main` is never committed to directly — it is the clean, user-facing branch managed exclusively by CI.

## Context

The repo mixes plugin source with dev tooling (specs, agent config, etc.). vim-plug clones the default branch (`main`) verbatim, so users currently receive all dev files. We need a branch that is always clean for users while keeping the full development context available during authoring.

## Considered Options

| Option | Pros | Cons |
|--------|------|------|
| **Dev branch + CI publish** | `main` always clean; dev tooling stays co-located | Requires CI setup; two branches to manage |
| Single `main` branch + `.gitattributes` | Simple | `export-ignore` doesn't apply to `git clone`; users still get dev files |
| Separate dev repo | Completely clean plugin repo | Splits context; harder to keep in sync |

## Consequences

- `dev` is the new default branch for development
- All PRs target `dev`, not `main`
- CI workflow (`.github/workflows/release.yml`) copies plugin files from `dev` to `main` on every push to `dev`
- `main` is set as a protected branch (no direct pushes)
